# AGENTS.md

This file provides guidance for AI coding agents working on the Ancela project.

## Project Overview

Ancela is an experimental, **single-owner** AI assistant that users interact with over
SMS. It connects to real-world services (calendar, email, contacts, budget, the web) and
adds limited autonomy (reminders, standing rules, scheduled tasks). It's built with:

- **.NET 10** with C# 13
- **Aspire** for cloud-native orchestration and Bicep infrastructure-as-code
- **Azure Functions** (isolated worker model, v4)
- **Azure Cosmos DB** for data storage
- **Azure Service Bus** for the message queues
- **Semantic Kernel** for AI/LLM orchestration
- **OpenAI (`gpt-5-mini`)** for the conversational model
- **Twilio** for SMS, **Microsoft Graph** (calendar/email/contacts), **YNAB** (budget),
  **Tavily** (web search), and **reMarkable** (owner-only document upload)

See `README.md` for the product overview, security/trust model, and setup. This file
focuses on internals for working in the code.

## Build & Run Commands

```bash
# Build the solution
dotnet build

# Run locally with Aspire (launches all services + dashboard)
dotnet run --project Ancela.AppHost

# Run tests
dotnet test

# Deploy to Azure (see README for full flow)
ANCELA_RESOURCE_PREFIX=your-prefix aspire publish
aspire deploy --clear-cache
```

## Code Style & Conventions

- Use **file-scoped namespaces**
- Use **primary constructors** for dependency injection
- Use **raw string literals** (`"""`) for multi-line strings
- Prefer **async/await** patterns throughout
- Use **nullable reference types** (enabled project-wide)
- Follow standard C# naming conventions (PascalCase for public members, _camelCase for private fields)
- Use **4 spaces for indentation** (enforced via .editorconfig)
- If an interface has a single implementation, include the interface definition at the top of the implementation file.

## Key Components

### Ancela.AppHost
- `AppHost.cs` - Aspire distributed application builder; defines storage, Service Bus queues (`chat-messages`, `reminders`, `standing-rules`, `scheduled-tasks`), Cosmos, OpenAI, and the Functions project, wiring secrets/env vars.
- `FixedNameInfrastructureResolver.cs` - Custom deterministic naming for Azure resources (uses `ANCELA_RESOURCE_PREFIX`).
- `aspire-output/` - Generated Bicep templates (regenerated by `aspire publish`).

### Ancela.FunctionApp
- `Program.cs` - Function app startup and DI (registers the Service Bus client and the agent module).
- `IncomingSms.cs` - HTTP trigger for Twilio webhooks; enqueues the message to the `chat-messages` Service Bus queue.
- `IncomingMessage.cs` - HTTP trigger for testing (non-Twilio); also enqueues to `chat-messages`.
- `ChatQueueProcessor.cs` - `ServiceBusTrigger` on `chat-messages`; routes to `ChatInterceptor`.
- `ReminderQueueProcessor.cs` / `StandingRuleQueueProcessor.cs` / `ScheduledTaskQueueProcessor.cs` - `ServiceBusTrigger` processors that run the due reminder / autonomous evaluation / scheduled task.

### Ancela.Agent
- `Agent.cs` - Core orchestration: builds a kernel per request via `IKernelFactory`, advertises the profile-allowed functions to the model, and runs `Chat`, `Onboard`, `EvaluateStandingRule`, and `PerformScheduledTask`.
- `ChatInterceptor.cs` - Handles session/access commands before the model sees them: `hello ancela`, `goodbye ancela`, and owner-only `invite`/`revoke` (with required TOTP step-up). Invite-only registration.
- `AuditFilter.cs` - `IFunctionInvocationFilter` that audit-logs every kernel function invocation.
- `AutonomousToolGuardFilter.cs` - `IFunctionInvocationFilter` that hard-denies any function call outside the profile's allow-list (and owner-only functions for non-owners), as defense-in-depth.
- `DependencyModule.cs` - Service registration for the agent module.

#### Services (`Services/` folder)
- `UserService.cs` - User profiles / access state in Cosmos (replaces the old session service).
- `HistoryService.cs` - Per-user conversation history in Cosmos.
- `OwnerService.cs` - Identifies the owner (by `OWNER_PHONE_NUMBER`).
- `TotpService.cs` - RFC 6238 owner step-up verification (`OWNER_TOTP_SECRET`).
- `SmsService.cs` - Twilio SMS integration.
- `MediaService.cs` - Handling of inbound MMS media.
- `AuditService.cs` - Cosmos-backed audit log (`IAuditLog` / `CosmosAuditLog`).

#### Semantic Kernel (`SemanticKernel/` folder)
- `KernelFactory.cs` - Builds a fresh `Kernel` per request for a given `KernelProfile` (`Chat`, `Onboarding`, `StandingRule`, `ScheduledTask`), loads plugins, and attaches the invocation filters. Onboarding loads only `RegistrationPlugin`.
- `KernelProfilePolicy.cs` - **Single source of truth** for which functions each profile may use. Autonomous profiles (`StandingRule`, `ScheduledTask`) are default-deny allow-lists; `Chat`/`Onboarding` are unrestricted. Also defines owner-only functions (`send_sms`, `send_email`, `create_calendar_event`, `send_to_remarkable`). Consumed both when advertising functions to the model and by `AutonomousToolGuardFilter`.

#### Plugins (`SemanticKernel/Plugins/` folder)
Each plugin pairs a `*Plugin` class (kernel functions) with a client/store/service for I/O.
- `MemoryPlugin/` - To-dos and knowledge (shared memory) in Cosmos.
- `ProjectsPlugin/` - Projects: longer-lived workspaces with notes and categorized entries.
- `GraphPlugin/` - Microsoft Graph: calendar, email, contacts.
- `YnabPlugin/` - YNAB budget/account/transaction reads.
- `WebPlugin/` - `web_search` (Tavily) and `web_fetch`.
- `SmsPlugin/` - `send_sms` (owner-only).
- `ReminderPlugin/` - One-time SMS reminders (store + scheduler).
- `StandingRulePlugin/` - Recurring conditions evaluated on a timer (store + scheduler).
- `ScheduledTaskPlugin/` - Recurring actions on a clock schedule (store + scheduler).
- `RegistrationPlugin/` - User onboarding (`register_user`).
- `RemarkablePlugin/` - `send_to_remarkable` (owner-only); renders text to PDF with QuestPDF and uploads via `Remarkable.Api.Client`. Requires `REMARKABLE_DEVICE_TOKEN` (obtained one-time via `rmpair`).

### Ancela.Cli
Read-only Spectre.Console TUI for auditing the deployed Cosmos data and minting the owner TOTP secret. Commands: `ping`, `list`, `show` (browse containers), and `enroll` (generate `OWNER_TOTP_SECRET` + QR).

### Ancela.Agent.Tests
xUnit tests covering agent capabilities and security: `AgentTodoTests`, `AgentKnowledgeTests`, `AgentGraphTests`, `AgentProjectsTests`, `AgentMediaTests`, `ChatInterceptorTests`, `SecurityTests`, `TotpServiceTests`, `MediaServiceTests`, `ScheduleCalculatorTests` (base class: `AgentTestBase`).

## Configuration

### Local Development
User secrets are set in `Ancela.AppHost` (see README for the full list), e.g.:
```bash
dotnet user-secrets set Parameters:openai-api-key      "..."
dotnet user-secrets set Parameters:twilio-account-sid  "..."
dotnet user-secrets set Parameters:graph-client-secret "..."
dotnet user-secrets set Parameters:ynab-access-token   "..."   # optional
```
Owner step-up uses `OWNER_TOTP_SECRET` (from `ancela enroll`); resource naming uses the
`ANCELA_RESOURCE_PREFIX` / `ANCELA_RESOURCE_GROUP` environment variables.

### Azure Deployment
- Secrets are prompted during `aspire deploy`.
- Cosmos DB uses role-based access; run `./configure_cosmos.sh` post-deployment to grant the
  role and create the database + containers.

## Cosmos DB Data Model

Single database (`anceladb`). All containers partition on `/agentPhoneNumber` (one value per
instance):
- **users** - User profiles and access state
- **todos** - To-do items (shared)
- **knowledge** - Knowledge entries (shared)
- **projects** - Projects and their entries (shared)
- **history** - Conversation history (per-user; filtered by `userPhoneNumber`)
- **reminders** - One-time reminders (per-user)
- **standing_rules** - Standing rules (per-user)
- **scheduled_tasks** - Scheduled tasks (per-user)
- **audit** - Audit log of function invocations and session events

## Important Patterns

### Semantic Kernel Integration
- `IKernelFactory.Create(profile)` builds a fresh `Kernel` per request with its own `Kernel.Data` (per-request `agentPhoneNumber` / `userPhoneNumber`), so identity is never shared across concurrent invocations.
- The model only ever sees the functions allowed by `KernelProfilePolicy` for the active profile (and, for non-owners, with owner-only functions removed) — advertised via `FunctionChoiceBehavior.Auto(functions:)`.
- Two `IFunctionInvocationFilter`s run on every kernel: `AuditFilter` (logging) and `AutonomousToolGuardFilter` (hard-deny backstop matching the same policy).
- Context is passed via `kernel.Data[]`.

### Session & Message Flow
1. Twilio webhook → `IncomingSms` enqueues to the `chat-messages` Service Bus queue and returns immediately.
2. `ChatQueueProcessor` dequeues → `ChatInterceptor`:
   - `hello ancela` → registers the owner, or onboards an already-invited user (others are silently dropped — invite-only).
   - owner `invite <number> <totp>` / `revoke <number> <totp>` → access management, gated by required TOTP step-up (fail-closed).
   - `goodbye ancela` → removes the account.
   - anything else → `Agent.Chat` (or `Agent.Onboard` if the profile is pending a name).
3. Autonomous work runs from the reminder / standing-rule / scheduled-task Service Bus queues via their processors, using the restricted kernel profiles.

## Testing Guidelines

- Use xUnit. Mock external services (Cosmos, Twilio, OpenAI/Graph/YNAB) — see `AgentTestBase`.
- Security-relevant behavior (profile allow-lists, owner-only gating, TOTP) has dedicated tests (`SecurityTests`, `TotpServiceTests`, `ChatInterceptorTests`) — keep them passing when touching those paths.

## Common Tasks

### Adding a New Semantic Kernel Plugin
1. Create a folder under `Ancela.Agent/SemanticKernel/Plugins/` with a `*Plugin` class (`[KernelFunction]` + `[Description]`) and a client/store for I/O.
2. Register the plugin and its client/store in `DependencyModule.cs`.
3. Inject the plugin into `KernelFactory` and add it to the relevant profile branch(es) in `Create`.
4. If any function should be callable by autonomous profiles, add it to the appropriate allow-list in `KernelProfilePolicy`; if it sends/mutates owner data, add it to `OwnerOnlyFunctions`.
5. Update the system prompt in `Agent.cs` if needed, and add tests.

### Adding a Capability to an Existing Plugin
1. Add a `[KernelFunction]` method to the plugin class.
2. Update `KernelProfilePolicy` if it needs autonomous or owner-only treatment.
3. Update the system prompt in `Agent.cs` if needed, and add tests in `Ancela.Agent.Tests/`.

### Modifying Infrastructure
1. Update `AppHost.cs` with new resources/references.
2. Re-run `dotnet run --project Ancela.AppHost` locally; `aspire publish` regenerates the Bicep in `aspire-output/`.
3. Deploy with `aspire deploy`.
